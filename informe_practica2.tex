\section{Introducción}
En la Práctica 1 se implementó la lectura de tramas NMEA (sentencia GGA) y la conversión de latitud/longitud a coordenadas UTM. En esta segunda práctica extendemos esa aplicación para que, además de calcular la posición, la represente sobre un mapa en pantalla.

El sistema desarrollado permite trabajar con múltiples mapas georreferenciados, concretamente tres escenarios: Campus Sur, INSIA y Vicálvaro. Cada imagen se calibra mediante coordenadas UTM de sus esquinas, permitiendo representar en tiempo real la posición del GPS sobre el mapa de forma coherente.

\section{Objetivo y entregables}
\textbf{Objetivo:} extender la aplicación anterior para mostrar en una GUI la posición suministrada por el receptor GPS sobre imágenes georreferenciadas (Campus Sur, INSIA y Vicálvaro), con una visualización similar a la de un navegador comercial.

\textbf{Entregables:}
\begin{itemize}[leftmargin=1.2em]
\item Código fuente de la aplicación (archivo \texttt{practica2.py}).
\item Imágenes base: \texttt{campus\_sur.jpg} y \texttt{gps\_insia.jpg}.
\item Documento PDF con la descripción del trabajo (este informe).
\end{itemize}

\section{Obtención de las imágenes (Google Earth Pro)}
Las imágenes utilizadas se obtuvieron mediante capturas de Google Earth Pro, manteniendo:

\begin{itemize}[leftmargin=1.2em]
\item Vista cenital (sin inclinación)
\item Norte orientado hacia arriba
\item Zoom fijo durante la captura
\end{itemize}

Una vez obtenidas las imágenes, se determinaron las coordenadas UTM de sus esquinas (superior izquierda e inferior derecha), que se emplean directamente para la georreferenciación.

\section{Georreferenciación (UTM $\rightarrow$ píxel)}

Para cada mapa se definen dos puntos de referencia:

\begin{itemize}[leftmargin=1.2em]
\item TL (\emph{top-left}): esquina superior izquierda
\item BR (\emph{bottom-right}): esquina inferior derecha
\end{itemize}

Estas esquinas se expresan directamente en coordenadas UTM (sistema WGS84, huso 30), lo que simplifica el proceso de georreferenciación.

La conversión de coordenadas UTM a píxel se realiza mediante interpolación lineal:

\[
x = \frac{E - E_{TL}}{E_{BR}-E_{TL}}\,(W-1), \qquad
y = \frac{N_{TL} - N}{N_{TL}-N_{BR}}\,(H-1)
\]

donde $(E,N)$ son las coordenadas UTM del GPS y $(W,H)$ el tamaño de la imagen en píxeles.

\section{Implementación}

La aplicación se ha desarrollado en Python, reutilizando el procesamiento de datos GPS de la práctica anterior. Se ha diseñado una estructura modular basada en una clase de configuración (\texttt{MapConfig}) que permite definir distintos mapas de forma independiente.

El usuario puede seleccionar el mapa a visualizar mediante argumentos en línea de comandos:

\begin{lstlisting}[style=pythonEstilo]
python practica2.py --map insia
python practica2.py --map campus
python practica2.py --map vicalvaro
\end{lstlisting}

Esto permite reutilizar la misma aplicación para distintos escenarios sin modificar el código.

\subsection{Lectura del GPS}
La lectura por puerto serie se hace en un hilo para no bloquear la GUI. Se filtran tramas GGA, se valida el checksum y, si el fix es válido, se actualiza la pantalla.

\subsection{Visualización tipo navegador}
En pantalla se muestran:
\begin{itemize}[leftmargin=1.2em]
\item Marcador de posición (punto) y etiqueta.
\item Traza simple del recorrido uniendo posiciones consecutivas.
\item Panel con hora UTC, calidad del fix, satélites, HDOP, altitud y coordenadas.
\end{itemize}

\section{Resultados y pruebas}
Se ha comprobado el correcto funcionamiento del sistema en los tres mapas implementados. La posición GPS se representa correctamente dentro del área georreferenciada, y el sistema detecta adecuadamente cuando el punto se encuentra fuera de la imagen.

La inclusión de múltiples mapas demuestra la flexibilidad del sistema desarrollado y su capacidad para adaptarse a diferentes entornos geográficos.

Por ahora la validación se ha hecho a nivel funcional (recepción de GGA, conversión UTM y representación en el mapa). Las pruebas finales con vehículo instrumentado en la pista del INSIA se dejarán para la sesión de evaluación, donde se podrá valorar la coherencia del trazado en movimiento real.
